# Conceptual Blend Template — Operational Reference

## Four-Space Architecture (Fauconnier & Turner)

A conceptual blend integrates elements from two input spaces into a blend that generates emergent structure — properties and relations that exist in neither input space alone.

## Template (fill all four spaces)

```
## Input Space 1 — [Domain A name]
Elements: [list key elements, roles, relations]
Structure: [what rules/constraints govern this domain]
Relevance: [why this domain is being blended into the target problem]

## Input Space 2 — [Domain B / Original problem]
Elements: [list key elements, roles, relations]  
Structure: [what rules/constraints govern this domain]
Note: this is typically the problem domain itself

## Generic Space — [Shared skeleton]
Shared structure: [what structural roles are common to both inputs?]
Cross-space mapping: [which elements in Input 1 correspond to which in Input 2?]
Note: generic space captures ABSTRACT relationships, not surface features

## Blended Space — [Name the blend]
Selective projection:
  - From Input 1: [which elements are projected?]
  - From Input 2: [which elements are projected?]
Operations applied:
  - Composition: [what new relations arise between projected elements?]
  - Completion: [what background knowledge activates from each input?]
  - Elaboration: [run the blend mentally — what happens when the scenario plays out?]

## Emergent Structure — CRITICAL SECTION
[What is true in this blend that was NOT true in either input space alone?]
[Emergent features must be genuinely new — not just inherited from one input]
[If no emergent structure: blend is not blend-worthy; skip and try a different pairing]
```

## Blend-Worthiness Test

A mapping is blend-worthy if ≥1 of the following holds:
- The combination produces emergent features (things true in the blend, not in either input)
- The mapping reveals a hidden structural isomorphism between domains
- The blend opens solution paths that neither domain suggested independently

**If no mapping passes the blend-worthiness test:** skip Step 7 (blend construction) and proceed directly to Step 8 (de Bono lateral fallback) if needed.

## Emergent Feature Tagging

For each identified emergent feature, tag as:
- `[EMERGENT-SOLUTION]` — directly points to a problem solution
- `[EMERGENT-FRAME]` — reframes the problem in a new way
- `[EMERGENT-CONSTRAINT]` — reveals a new constraint previously invisible
- `[EMERGENT-ANALOGY]` — suggests an analogical transfer path

## Examples of Emergent Structure

- Buddhist monk riddle: Two monks (Input 1: ascending, Input 2: descending) → Blend: both on same day → Emergent: they MUST meet (neither input alone implies this)
- Computer virus: Computer program (Input 1) + Biological virus (Input 2) → Blend: self-replicating destructive code → Emergent: immune system analogies, infection metaphors, quarantine solutions
- "Desktop" metaphor: Office (Input 1) + Screen (Input 2) → Blend: digital workspace → Emergent: dragging, filing, trashing documents

Write under `<conceptual_blend>` sub-element in the stage output.
Only write this section when Step 6 verification tagged ≥1 mapping as blend-worthy.
If no emergent features arise: note "blend attempted, no emergent structure found" and do NOT fabricate emergent properties.
